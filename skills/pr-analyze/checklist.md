# PR 审查清单

> 语言无关。适用于任何语言、任何框架的 PR 审查。
> 配合 `/pr-analyze` skill 使用。

---

## 使用方式

审查分两轮：

- **Pass 1 — CRITICAL**：数据安全、并发安全、信任边界、数据完整性。必须先完成。
- **Pass 2 — INFORMATIONAL**：可维护性、错误处理、性能、兼容性、测试。在 Pass 1 之后进行。

Pass 1 之前，先执行 **数据流追踪**（见下方）。

---

## 数据流追踪（前置步骤）

在执行 checklist 之前，从 diff 中识别所有数据表示变换点，然后用完整源码追踪每个变换的 roundtrip。

### 识别变换点

从 diff 中找所有做格式转换的函数/方法（输入一种类型，输出另一种类型）。常见模式：

- 序列化/反序列化（对象 ↔ JSON、对象 ↔ 数据库行、对象 ↔ 协议消息）
- 层间适配器（外部 API 类型 ↔ 内部模型、UI 模型 ↔ 业务模型）
- 编解码（压缩/解压、加密/解密、编码/解码）

### 追踪 roundtrip

对每个变换点，用克隆的完整源码追踪完整路径：

1. **写入端**：输入数据 → 变换函数 → 持久化存储（数据库/文件/缓存）
2. **读出端**：持久化存储 → 反向变换函数 → 输出数据
3. **验证一致性**：写入时存了什么，读出时是否完整还原

### 必须检查

- 数据经变换后是否丢失字段？（写入时有，存储后没有，读出时无法还原）
- 存储操作是否静默丢弃部分输入？
- 读出时是否假设了写入时没有保证的不变量？
- 并发读写时中间状态是否一致？

### 报告格式

每个发现标注：

```
[SEVERITY] (confidence: N/10) 文件路径:行号 — 问题描述
  影响: 具体说明
  建议: 修复方案
```

## Pass 1 — CRITICAL（阻塞合并）

### 1.1 数据安全

- 字符串拼接构建查询（SQL/命令/路径/URL）是否有注入风险？使用参数化查询或转义。
- 读-检查-写模式是否缺少原子性保证？应使用原子条件更新。
- 绕过模型/层验证直接写入数据库是否会导致数据不一致？
- 循环中的逐项查询是否可批量化？注意 N+1 问题。

### 1.2 并发安全

- 共享状态的读-检查-写操作是否缺少互斥或原子保证？
- 异步操作与同步操作并发访问同一资源时是否有竞态窗口？
- 状态转换是否使用原子操作（而非先读后写）？
- 并发写入是否可能导致重复创建？

### 1.3 信任边界

- 外部输入（用户输入、AI 输出、第三方 API 响应）是否在持久化前做了格式验证？
- 外部生成的 URL 是否在请求前校验了目标（防 SSRF）？
- 外部内容是否在不经过清理的情况下被渲染为富文本（XSS）？
- AI 生成的结构化输出是否在写入前检查了类型和结构？
- AI 输出存储到知识库/向量库前是否做了清理（防存储型注入）？

### 1.4 命令/代码注入

- 是否存在用外部输入构建 shell 命令的情况？使用参数数组代替字符串拼接。
- 是否存在对外部输入执行 `eval`/`exec` 的情况？需要沙箱化。
- 是否存在用外部输入构建文件路径的情况？校验路径遍历。

### 1.5 数据完整性

- 新增的枚举值/状态/类型常量是否在所有消费端都被处理？
  - 追踪方法：搜索已有的同类值，找到所有引用处，逐一检查新值是否被处理
  - 需要**读取 diff 之外的代码**来完成此检查
- case/switch/match 分支是否有遗漏的新值？是否有 fallback 分支？
- 过滤条件/白名单/允许列表是否包含了新增的值？

---

## Pass 2 — INFORMATIONAL（建议改进）

### 2.1 可维护性

- 核心文件是否超过 600 行？建议按职责拆分。
- 一个模块是否承担了过多不相关功能？
- 新增代码是否可以复用已有基础设施而非重写？

### 2.2 错误处理

- 是否存在静默吞掉错误的 catch/except/recover？
- 循环终止是否仅依赖"无进展"条件而缺少硬性上限？
- 关键操作的失败路径是否会导致不一致状态？
- 错误信息是否足够帮助定位问题？

### 2.3 性能

- 循环中是否有可批量化为单次查询的逐项操作？
- 是否有可能的内存泄漏（资源未释放、缓存无限增长、goroutine 泄漏）？
- 大数据集是否使用了流式处理而非全量加载？

### 2.4 兼容性

- 接口签名变更是否向后兼容？
- 配置变更是否有合理的默认值？
- 数据结构变更是否考虑了迁移路径？
- 新增依赖是否会引入版本冲突？

### 2.5 测试

- 关键逻辑路径是否有对应测试？
- 边界条件和异常路径是否被测试覆盖？
- 并发场景是否有测试？
- 测试是否覆盖了数据 roundtrip（序列化 → 存储 → 反序列化）？

### 2.6 范围漂移

- PR 的实际改动是否超出了声明意图？
- 是否有"顺便"改了不相关的功能？
- 是否有声明了但没实现的需求？

## 严重度与置信度

### 严重度

| 级别 | 标准 | 报告位置 |
|------|------|---------|
| CRITICAL | 运行时错误/安全漏洞/数据丢失，正常场景可触发，无安全网 | 主报告，阻塞合并 |
| WARNING | 异常行为，但存在安全网或影响有限 | 主报告，建议修复但不阻塞 |
| INFORMATIONAL | 设计偏好、可维护性建议 | 主报告，non-blocking |

**判定口诀**：删除这行代码会引发 bug 且无安全网 → CRITICAL。有安全网兜底 → WARNING。"我会选择不同的做法" → INFORMATIONAL。

### 置信度（1-10）

每个发现必须标注置信度：

| 分数 | 含义 | 显示规则 |
|------|------|---------|
| 9-10 | 已通过阅读具体代码验证，能指出确切的 bug 路径 | 正常显示 |
| 7-8 | 高置信度模式匹配，很可能正确 | 正常显示 |
| 5-6 | 中等置信度，可能是误报 | 标注"需验证" |
| 3-4 | 仅怀疑，证据不足 | 仅放附录 |
| 1-2 | 猜测 | 仅在 P0 级问题时报告 |

**发现格式**：

```
[CRITICAL] (confidence: 8/10) path/to/file:42 — 描述
  影响: ...
  建议: ...
```

## DO NOT flag（抑制规则）

不要报告以下内容：

- **已在 diff 中解决的问题** — 读完整 diff 后再评论，不要标记已经修好的东西
- **纯风格偏好** — 命名方式、代码组织偏好、"我会用不同的设计模式"
- **"建议加注释解释为什么"** — 注释会过时，阈值/常量是经验调参的结果
- **冗余但无害的代码** — 提升可读性的冗余是合理的
- **断言可以更严格** — 已覆盖行为即可，不需要每个断言都精确到最小范围
- **阈值/常量的具体数值** — 这些是经验调参的结果，不评价值的选择
- **无害的 no-op 操作** — 不影响正确性的空操作
- **测试同时验证多个守卫条件** — 测试不需要隔离每个守卫
